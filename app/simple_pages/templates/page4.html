What is CI/CD?

Continuous Integration is a software development practice where members of a team integrate their work frequently, usually each person integrates at least daily - leading to multiple integrations per day. Each integration is verified by an automated build (including test) to detect integration errors as quickly as possible.

Continuous Delivery is a software development discipline where you build software in such a way that the software can be released to production at any time.You achieve continuous delivery by continuously integrating the software done by the development team, building executables, and running automated tests on those executables to detect problems. Furthermore you push the executables into increasingly production-like environments to ensure the software will work in production

How my project is setup to use Github Actions

GitHub Actions makes it easy to automate all your software workflows, now with world-class CI/CD. Build, test, and deploy your code right from GitHub. Make code reviews, branch management, and issue triaging work the way you want.

In the github actions I have added repositories secrets that help me connect with Heroku and Docker when ever my code is updated and committed with any new changes. In these secrets I have set DOCKER_PASSWORD, DOCKER_USERNAME and HEROUKU_API_KEY.

How I can review code using development server

There are two seperate workflow files 1. dev.yml and 2.prod.yml. The development workflow deploys the app on development server whenever a pull request is made. This helps in keeping the production server free of bugs and always up and running. Whenever someone implements a new feature on new branch and makes a pull request, we can review what changes he/she made and check if those changes are not breaking the code by testing it on development server and checking their test cases integrity. If the code is clean and bug free then we can merge pull request to master plan which will deploy the app on development server.

Deploy master to production server

The prod.yml workflow file has the information which helps us deploy master to production server. Whenever a code is committed and pushed on master branch or any merge pull request is made the app will get deployed on production server. Before deploying on production server it will run all the test cases and if and only if all the test cases pass then it will be deployed on heroku production app and on docker hub

Deploy an image of project to DockerHub when master updated

The prod.yml workflow file has the information which helps us deploy master to docker container. Whenever master is updated the workflow will run pytests, setup the python environment, install the dependencies, and install all the required packages mentioned in requirements.txt file. Then it will login to DockerHub account using the DOCKER_USERNAME and DOCKER_PASSWORD mentioned in github actions secrets and deploy the latest image in the mentioned Dcoker repository under image tag
